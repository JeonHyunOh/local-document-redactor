# 파일명·폴더명 키워드 제거 — 설계

- 작성일: 2026-07-13
- 상태: 승인됨 (구현 대기)
- 대상 레포: `local-document-redactor`

## 배경 / 문제

현재 도구는 **파일 내용만** 검사·삭제한다(Excel 셀 값, PDF 텍스트). 그러나 파일명·폴더명에
민감 키워드가 들어 있으면(예: `포스코_구형흑연폐수설비_설계.xlsx`) 내용을 아무리 지워도
**이름을 통해 정보가 그대로 유출**된다. 결과물은 원본 이름에 `_edited`/`_redacted` 접미사만
붙이므로 키워드가 파일명에 남는다.

또한 **내용은 깨끗한데 파일명에만 키워드가 있는 파일**은 지금 어느 모드에서도 처리되지 않는다.
배치는 `only_with_matches=True`가 내용 매치 0건이면 스킵하고, 단일 파일은 "발견된 키워드가
없습니다"로 종료해 다운로드조차 만들지 않는다. 이것이 누출 위험이 가장 큰 케이스다 — 내용을
지울 게 없다는 이유로 **파일명의 키워드가 그대로 남는다.**

## 목표

등록된 키워드를 **파일명과 폴더명(경로 세그먼트)** 에서도 제거한다. 단일 파일·폴더 출력본·
제자리 교체 세 가지 저장 모드 모두에 적용한다.

## 비목표 (범위 밖)

- 파일명에서의 "구절→단어 분해" 삭제. (사용자 결정: **등록 키워드 문자열 그대로 부분 제거**)
- 파일 **내용** 처리 방식 변경. 기존 동작 유지.
- 스캔 루트 폴더 자체의 이름 변경. (아래 안전장치 참조)

## 핵심 결정 (사용자 승인 완료)

1. **매칭 방식**: 이름에서는 등록 키워드를 **부분문자열로 그대로 제거**한다. 내용 검색이
   '정확히 일치(EXACT)' 모드여도 **이름은 항상 부분 제거**로 처리한다. `case_sensitive`는
   이름에도 동일 적용.
2. **이름-only 매치 처리**: 내용이 깨끗해도 **이름에 키워드가 있으면 처리 대상에 포함**한다.
   (단일 파일 = 정리된 이름의 원본 복사본 다운로드 제공, 출력본 모드 = 정리된 이름으로 복사,
   제자리 모드 = rename) — **세 모드 예외 없이.**
3. **적용 범위**: 세 저장 모드 모두 — 단일 파일 다운로드명 / 폴더 출력본+zip / 제자리 교체.
4. **안전장치**: 스캔 **루트 폴더 자체는 rename하지 않는다**(사용자가 입력한 대상 경로가
   바뀌면 혼란·참조 깨짐). 하위 파일·폴더만 대상. 루트 이름에 키워드가 있으면 그대로 두고
   경고를 표시한다.

## 원칙 — 이름 검사는 내용 검사와 완전히 독립이다

**핵심 불변식**: 파일명·폴더명의 키워드 검사·제거는 **파일 내용에 키워드가 있든 없든** 항상
독립적으로 수행한다. 어떤 파일도 "내용이 깨끗하다"는 이유로 이름 검사를 건너뛰지 않는다.

각 파일/폴더는 두 축으로 독립 판정된다:

| | 내용에 키워드 있음 | 내용 깨끗 |
|---|---|---|
| **이름에 키워드 있음** | 내용 편집 **+** 이름 정리 | **이름만 정리**(내용 무수정) |
| **이름 깨끗** | 내용 편집만 | 처리 안 함(건너뜀) |

- 폴더는 내용 개념이 없으므로 **이름 검사만** 적용한다.
- 이 매트릭스가 세 모드 모두의 "처리 대상" 판정 기준이다. 기존 `only_with_matches`(내용 매치
  기준 단일 축)를 **`내용 매치 OR 이름 매치`** 로 대체하는 것이 이 기능의 핵심 변경점이다.

## 아키텍처

기존 계층 경계를 유지한다. UI(`app.py`)에는 로직을 넣지 않고, 이름 정리는 순수 로직 계층에
새 모듈로 추가한 뒤 서비스 계층에서 호출한다.

### 신규 모듈: `src/document_redactor/name_redactor.py` (순수 로직)

`keyword_matcher`에만 의존하는 파일 I/O 없는 순수 함수 계층.

```
redact_filename(name: str, keywords: list[str], case_sensitive: bool) -> str
    # 파일명. 마지막 확장자(suffix)는 보존하고 stem에서만 키워드 제거.

redact_segment(name: str, keywords: list[str], case_sensitive: bool) -> str
    # 폴더명(확장자 개념 없음). 이름 전체에서 키워드 제거.

name_contains_keyword(name: str, keywords: list[str], case_sensitive: bool) -> bool
    # 이름-only 매치 판정용.

unique_name(candidate: str, taken: set[str]) -> str
    # 같은 디렉터리 내 충돌 시 stem 뒤에 _1, _2 … 부여(확장자 유지).
```

**정리(tidy) 규칙** (제거 후 남는 잔재 처리):
- 연속된 구분자(공백 `_` `-`) → 하나로 축약.
- 앞뒤의 공백 `_` `-` `.` 트리밍.
- 결과 stem이 비면 `redacted`로 대체.
- 확장자는 항상 원본 유지.

예시:
| 입력 | 등록 키워드 | 출력 |
|---|---|---|
| `포스코_구형흑연폐수설비_설계.xlsx` | `포스코`, `구형흑연폐수설비` | `설계.xlsx` |
| `posco.pdf` | `posco` | `redacted.pdf` |
| `2026_포스코_보고서.xlsx` | `포스코` | `2026_보고서.xlsx` |

### 재사용
기존 `keyword_matcher.remove_keywords` / `normalize_keywords`를 재사용한다. 이름 정리는
그 위에 tidy·suffix·collision 로직만 얹는다.

## 모드별 동작

### 1. 단일 파일 (`app.py: render_single_file`)
- 디스크 원본·임시 파일은 건드리지 않는다. **다운로드 파일명 문자열만** 정리.
- **내용 매치가 있는 경우**(기존 흐름): 편집본 생성 후 `download_button(file_name=...)`에
  `redact_filename(업로드파일명, …)` 결과를 넘긴다. (작업 로그 파일명도 동일 정리.)
- **내용 매치 0건 + 업로드 파일명에 키워드가 있는 경우**(신규, 이름-only): 현재는
  "발견된 키워드가 없습니다"로 종료해 **파일명 누출을 방치**한다. 이를 보완한다 —
  - 내용은 손대지 않고 **업로드 원본 바이트 그대로**, `redact_filename`으로 정리한 파일명의
    다운로드 버튼을 제공한다.
  - "내용에는 키워드가 없지만 **파일명**에 키워드가 있어 파일명만 정리했습니다" 안내 표시.
- **내용·이름 모두 깨끗**: 기존대로 "발견된 키워드가 없습니다".
- 어느 경우든 다운로드 버튼 위에 `원본명 → 정리명` 캡션 표시(이름 변경이 있을 때).

### 2. 폴더 → 별도 출력 폴더 + zip (`batch_service.batch_edit`)
- 각 대상 파일의 출력 경로를 만들 때 **부모 폴더 세그먼트 전부 + 파일명**을 정리한다.
- 대상 판정을 `내용 매치 OR 이름 매치`로 확장:
  - 내용 매치 있음 → 기존대로 `file_service.apply_edit` 후 정리된 경로에 저장.
  - 내용 깨끗 + 이름(파일명 또는 상위 폴더명) 매치 → `shutil.copy2`로 정리된 경로에 복사.
  - 둘 다 없음 → 건너뜀(기존과 동일).
- 같은 출력 디렉터리 내 이름 충돌은 `unique_name`으로 해소.
- `BatchEditItem`에 정리된 출력 상대경로를 담을 선택 필드 추가(§데이터 모델).

### 3. 폴더 → 제자리 교체 (`batch_service.batch_edit_in_place`)
2단계로 처리한다.

**Phase 1 — 내용 편집(기존 유지)**: 임시 폴더에 편집본 생성 → 재검증 → 통과 시에만
백업 후 제자리 교체. 실패 파일 원본 유지.

**Phase 2 — 이름 rename(신규)**:
1. rename 시작 전 `백업루트/_rename_log.txt`를 열고 `이전상대경로 → 새상대경로`를 기록한다
   (수동 복구용).
2. **파일 먼저** rename: 이름에 키워드가 있는 스캔 대상 파일을 정리명으로 rename.
   - rename 전, 해당 파일이 Phase 1에서 백업되지 않았다면(이름-only 케이스) **원본을 백업루트에
     먼저 복사**한 뒤 rename.
   - 충돌 시 `unique_name`.
3. **폴더는 그다음, 깊은 곳부터(bottom-up)** rename: 이름에 키워드가 있는 하위 디렉터리를
   정리명으로 rename. 깊은 경로부터 처리해 상위 rename이 하위 경로를 무효화하지 않게 한다.
   - **스캔 루트 자체는 제외.**
   - 충돌 시 `unique_name`.
4. 모든 rename을 `_rename_log.txt`에 남기고, 결과/로그에 변경 건수를 집계.

## 데이터 모델 (`models.py`)
- `BatchEditItem`에 선택 필드 추가:
  - `renamed_to: str | None = None` — 정리된 출력/최종 상대경로(변경 없으면 None).
- 기존 필드·계약은 변경하지 않는다.
- 단일 파일 다운로드명은 모델 필드로 두지 않는다. `app.py`가 `name_redactor.redact_filename`
  (순수 함수)을 직접 호출해 표시·다운로드명에 사용한다 — 기존에 `keyword_matcher.normalize_keywords`를
  직접 호출하는 것과 동일한 패턴(불필요한 모델 확장 회피).

## 재검증 (안전 불변식 유지)
- 내용 재검증은 기존대로 수행한다.
- **이름 재검증 추가**: 최종 산출 이름/경로 세그먼트에 키워드가 남아 있는지 확인한다.
  - 남는 정상 케이스: 스캔 루트 이름(정책상 미변경), 충돌 회피로 부득이 남는 경우.
  - UI에 "이름에 키워드 잔존"을 별도로 보고한다.

- 단일 파일: 다운로드 버튼 위 `원본명 → 정리명` 캡션. 변경 없으면 생략. 내용 매치 0건이라도
  파일명에 키워드가 있으면 "파일명만 정리된 원본 복사본" 다운로드를 제공하고 그 사유를 안내.
- 폴더 결과표: "변경된 이름"(old → new) 열 추가 + "이름 변경 수" 지표.
- 제자리 모드: rename 로그 위치(`백업루트/_rename_log.txt`) 안내.
- 루트 이름에 키워드가 있으면 경고 표시.

## 안전 불변식 준수 확인
- 원본 미덮어쓰기: 단일·출력본 모드는 원본 무수정. 제자리 모드는 rename 전 **항상 백업**하고
  `_rename_log.txt`로 복구 경로 제공.
- 부분 실패를 정상처럼 넘기지 않음: rename 충돌·실패는 격리해 보고, 배치는 계속.
- 예외를 삼키지 않음: 파일별 오류 격리 시 사유를 `error`에 기록.
- 경로는 `pathlib.Path`로만 처리.

## 미해결/후속 (범위 밖 기록)
- 심볼릭 링크·바로가기가 rename된 파일을 참조하던 경우의 자동 복구는 하지 않음(로그로 안내).
- Windows 파일 잠금(열려 있는 문서) 중 rename 실패는 파일별 오류로 격리.
